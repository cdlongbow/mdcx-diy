# 代码审查标准

这份文档告诉你 MDCx 项目审查代码时要达到什么标准。新手照着做就行。

## 代码质量标准

### 功能对不对

#### 必须做到

- 代码得把所有需求功能都实现了
- 边界情况（比如输入为空、数字为 0）要处理好
- 出错的时候要有妥善处理，不能让程序直接崩溃
- 不能有明显逻辑错误

#### 建议做到

- 代码写得简洁优雅
- 别为了"以后可能用"而过度设计
- 代码要容易看懂

### 好不好读

#### 必须做到

- 变量名和函数名要让人一看就懂是干嘛的
- 一个函数只做一件事
- 代码结构清楚，不绕来绕去
- 复杂的逻辑要写注释解释

#### 建议做到

- 尽量让代码自己就能说清楚，少依赖注释
- 注释要写得有用，别写废话
- 别在代码里直接写魔法数字和神秘字符串

### 好不好维护

#### 必须做到

- 代码要容易修改
- 代码要容易扩展新功能
- 代码要容易写测试
- 别复制粘贴重复代码

#### 建议做到

- 代码尽量模块化，各管各的
- 适当使用设计模式
- 尽量遵循 SOLID 原则

## 性能标准

#### 必须做到

- 不能有明显跑得慢的地方
- 内存、CPU 这些资源要用得合理
- 不能有内存泄漏
- 文件句柄、网络连接用完要关

#### 建议做到

- 数据库查询要优化，别一次查太多
- 网络请求要优化，别来回发太多
- 重复计算的结果可以缓存起来

## 安全标准

#### 必须做到

- 用户输入的东西必须验证，不能直接信任
- 要防 SQL 注入
- 要防 XSS（跨站脚本攻击）
- 要防路径穿越攻击
- API 密钥不能直接写在代码里
- 密码不能打到日志里

#### 建议做到

- 敏感数据存的时候要加密
- 权限给最小够用的就行
- 安全事件要记录日志方便审计

## 测试标准

#### 必须做到

- 核心功能要有单元测试
- 关键流程要有集成测试
- 测试要能独立运行，不依赖其他测试
- 测试代码也要好维护

#### 建议做到

- 测试覆盖率 >= 80%
- 核心模块覆盖率 >= 90%
- 最好有性能测试
- 最好有安全测试

## 文档标准

#### 必须做到

- 对外公开的 API 要写文档字符串
- 复杂逻辑要写注释
- README 要更新
- 变更日志要更新

#### 建议做到

- 注释要准确有用
- 文档要完整，别漏东漏西
- 文档最好有使用示例

## 兼容性标准

#### 必须做到

- 代码要在 Windows、macOS、Linux 上都能跑
- 代码要兼容项目指定的 Python 版本
- 配置文件要向后兼容，旧的配置还能用
- 数据格式要向后兼容

#### 建议做到

- 尽量兼容旧版本的依赖
- 有不兼容的改动要提供迁移指南

## 代码风格标准

#### 必须做到

- 代码要能通过 ruff 检查
- 遵循 PEP 8 规范
- 代码格式要统一

#### 建议做到

- 变量和函数加上类型注解
- 字符串格式化用 f-string
- 删掉没用到的代码

## 审查级别

审查的时候按这几个层面来看：

### 1. 功能审查

- 功能实现得对不对
- 边界情况处理了没
- 错误处理了没

### 2. 质量审查

- 代码风格好不好
- 代码结构清不清楚
- 代码好不好读

### 3. 性能审查

- 有没有性能瓶颈
- 资源用得合不合理
- 算法复杂度高不高

### 4. 安全审查

- 输入有没有验证
- 权限控制到不到位
- 敏感信息处理妥不妥

### 5. 测试审查

- 测试覆盖了主要代码没
- 测试写得好不好
- 测试好不好维护

## 审查结果

### 通过

所有"必须做到"的条目都达标了。

### 有条件通过

有一些"建议做到"的条目没满足，需要修一下再合。

### 不通过

有"必须做到"的条目没满足，必须修完重新审查。
