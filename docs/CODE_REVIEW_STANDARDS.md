# 代码审查标准

本文档定义 MDCx 代码审查的质量标准。

## 代码质量标准

### 功能性

#### 必须满足

- 代码必须实现所有需求功能
- 边界情况必须得到正确处理
- 错误处理必须完善
- 不能有明显的逻辑错误

#### 建议满足

- 代码应该优雅简洁
- 避免过度设计
- 代码应该易于理解

### 可读性

#### 必须满足

- 变量和函数命名必须清晰
- 函数职责必须单一
- 代码结构必须清晰
- 复杂逻辑必须有注释

#### 建议满足

- 代码应该自我解释
- 注释应该恰当且有用
- 应该避免魔法数字和字符串

### 可维护性

#### 必须满足

- 代码必须易于修改
- 代码必须易于扩展
- 代码必须易于测试
- 没有明显的代码重复

#### 建议满足

- 代码应该模块化
- 应该使用设计模式
- 应该遵循 SOLID 原则

## 性能标准

### 必须满足

- 不能有明显的性能瓶颈
- 资源使用必须合理
- 不能有内存泄漏
- 不能有资源泄漏

#### 建议满足

- 应该优化数据库查询
- 应该优化网络请求
- 应该缓存计算结果

## 安全标准

### 必须满足

- 用户输入必须验证
- SQL 注入必须防护
- XSS 必须防护
- 路径穿越必须防护
- API 密钥不能硬编码
- 密码不能记录日志

#### 建议满足

- 敏感数据应该加密存储
- 应该使用最小权限原则
- 应该审计安全事件

## 测试标准

### 必须满足

- 核心功能必须有单元测试
- 关键路径必须有集成测试
- 测试必须能够独立运行
- 测试必须可维护

#### 建议满足

- 测试覆盖率应该 >= 80%
- 核心模块覆盖率应该 >= 90%
- 应该有性能测试
- 应该有安全测试

## 文档标准

### 必须满足

- 公共 API 必须有文档字符串
- 复杂逻辑必须有注释
- README 必须更新
- 变更日志必须更新

#### 建议满足

- 注释应该准确且有用
- 文档应该完整
- 文档应该有示例

## 兼容性标准

### 必须满足

- 代码必须跨平台兼容
- 代码必须兼容指定 Python 版本
- 配置必须向后兼容
- 数据格式必须向后兼容

#### 建议满足

- 应该兼容旧版本依赖
- 应该提供迁移指南

## 代码风格标准

### 必须满足

- 代码必须通过 ruff 检查
- 必须遵循 PEP 8 规范
- 必须使用一致的格式

#### 建议满足

- 应该使用类型注解
- 应该使用 f-string
- 应该避免不必要的代码

## 审查级别

### 1. 功能审查

- 审查代码是否实现预期功能
- 审查边界情况处理
- 审查错误处理

### 2. 质量审查

- 审查代码风格
- 审查代码结构
- 审查代码可读性

### 3. 性能审查

- 审查性能瓶颈
- 审查资源使用
- 审查算法复杂度

### 4. 安全审查

- 审查输入验证
- 审查权限控制
- 审查敏感信息处理

### 5. 测试审查

- 审查测试覆盖
- 审查测试质量
- 审查测试可维护性

## 审查结果

### 通过

代码符合所有必须满足的标准。

### 有条件通过

代码存在建议满足的问题，需要修复。

### 不通过

代码存在必须满足的问题，必须修复后重新审查。
