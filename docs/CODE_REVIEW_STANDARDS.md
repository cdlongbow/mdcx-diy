# 代码审查标准

本文档定义 MDCx 项目代码审查的质量标准。

## 代码质量标准

### 功能

#### 必须做到

- 代码实现所有需求功能
- 边界情况（输入为空、数字为 0 等）处理妥当
- 错误妥善处理，程序不能直接崩溃
- 没有明显逻辑错误

#### 建议做到

- 代码简洁优雅
- 避免过度设计
- 代码容易看懂

### 可读性

#### 必须做到

- 变量名和函数名一看就懂是干嘛的
- 一个函数只做一件事
- 代码结构清楚
- 复杂逻辑写注释

#### 建议做到

- 让代码自己能说清楚，少依赖注释
- 注释写得有用
- 不在代码里写魔法数字和神秘字符串

### 可维护性

#### 必须做到

- 代码容易修改
- 代码容易扩展
- 代码容易写测试
- 不复制粘贴重复代码

#### 建议做到

- 代码模块化
- 适当使用设计模式
- 遵循 SOLID 原则

## 性能标准

#### 必须做到

- 没有明显性能瓶颈
- 内存、CPU 使用合理
- 没有内存泄漏
- 文件句柄、网络连接用完要关闭

#### 建议做到

- 优化数据库查询
- 优化网络请求
- 缓存重复计算结果

## 安全标准

#### 必须做到

- 用户输入必须验证
- 防 SQL 注入
- 防 XSS（跨站脚本攻击）
- 防路径穿越攻击
- API 密钥不能直接写在代码里
- 密码不能打到日志里

#### 建议做到

- 敏感数据存储时加密
- 最小权限原则
- 安全事件记录日志

## 测试标准

#### 必须做到

- 核心功能有单元测试
- 关键流程有集成测试
- 测试独立运行，不依赖其他测试
- 测试代码好维护

#### 建议做到

- 测试覆盖率 >= 80%
- 核心模块覆盖率 >= 90%
- 有性能测试
- 有安全测试

## 文档标准

#### 必须做到

- 对外公开的 API 写文档字符串
- 复杂逻辑写注释
- README 更新
- 变更日志更新

#### 建议做到

- 注释准确有用
- 文档完整
- 文档有使用示例

## 兼容性标准

#### 必须做到

- 支持 Windows、macOS、Linux
- 兼容项目指定的 Python 版本
- 配置文件向后兼容
- 数据格式向后兼容

#### 建议做到

- 尽量兼容旧版本依赖
- 不兼容改动提供迁移指南

## 代码风格标准

#### 必须做到

- 通过 ruff 检查
- 遵循 PEP 8 规范
- 代码格式统一

#### 建议做到

- 变量和函数加类型注解
- 字符串格式化用 f-string
- 删除没用的代码

## 审查级别

### 1. 功能审查

- 功能实现是否正确
- 边界情况处理了没
- 错误处理了没

### 2. 质量审查

- 代码风格
- 代码结构
- 代码可读性

### 3. 性能审查

- 性能瓶颈
- 资源使用
- 算法复杂度

### 4. 安全审查

- 输入验证
- 权限控制
- 敏感信息处理

### 5. 测试审查

- 测试覆盖
- 测试质量
- 测试可维护性

## 审查结果

### 通过

所有"必须做到"的条目都达标。

### 有条件通过

有一些"建议做到"的条目没满足，需修改后合并。

### 不通过

有"必须做到"的条目没满足，必须修完重新审查。
