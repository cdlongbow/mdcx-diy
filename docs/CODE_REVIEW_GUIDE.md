# 代码审查指南

本文档提供 MDCx 代码审查的流程和最佳实践。

## 代码审查流程

### 1. 创建 Pull Request

在提交代码审查前：

1. 确保代码已通过本地测试
2. 运行 `uv run ruff check .` 检查代码风格
3. 运行 `uv run pytest tests/` 运行测试
4. 更新相关文档
5. 编写清晰的 PR 描述

### 2. 代码审查检查点

审查者应关注以下方面：

- **功能性**：代码是否实现了预期功能
- **代码质量**：代码是否清晰、可维护
- **性能**：是否存在性能问题
- **安全性**：是否存在安全隐患
- **测试**：是否有足够的测试覆盖
- **文档**：文档是否完整准确

### 3. 提供反馈

反馈应该：

- 具体明确
- 建设性
- 尊重他人
- 提供改进建议

### 4. 处理反馈

作者应该：

- 认真阅读所有反馈
- 询问不清楚的地方
- 及时修复问题
- 保持开放心态

## 审查最佳实践

### 对审查者

1. **及时审查**：尽量在 24 小时内完成审查
2. **全面审查**：不要只关注部分代码
3. **积极反馈**：提供建设性的建议
4. **保持礼貌**：尊重他人的劳动

### 对作者

1. **小步提交**：每次 PR 不要太大
2. **清晰描述**：提供详细的 PR 描述
3. **响应及时**：及时回复审查意见
4. **持续改进**：不断提高代码质量

## 审查工具

### 自动化检查

- **ruff**：代码风格检查
- **pytest**：单元测试
- **pytest-cov**：测试覆盖率
- **CI/CD**：自动运行检查

### 手动检查

- **代码走查**：逐行检查代码
- **功能测试**：手动测试功能
- **性能测试**：测试性能瓶颈
- **安全审查**：检查安全问题

## 审查模板

### PR 描述模板

```markdown
## 变更概述

简要描述本次变更的目的和内容

## 变更类型

- [ ] 新功能
- [ ] Bug 修复
- [ ] 重构
- [ ] 文档更新
- [ ] 性能优化
- [ ] 其他

## 影响范围

列出受影响的模块和文件

## 测试

描述如何测试本次变更

## 相关 Issue

关联相关的 Issue 或 PR

## 截图（如适用）

添加相关截图
```

### 审查评论模板

```markdown
### 问题

描述发现的问题

### 建议

提供改进建议

### 示例代码（如适用）

```python
# 示例代码
```
```

## 审查标准

### 必须修复（Must Fix）

- 功能性错误
- 安全隐患
- 性能问题
- 测试失败

### 建议修复（Should Fix）

- 代码风格问题
- 可读性问题
- 文档缺失
- 测试不足

### 可选修复（Nice to Fix）

- 命名优化
- 注释改进
- 小的优化

## 常见问题

### Q: 审查需要多长时间？

A: 小型 PR 1-2 小时，大型 PR 可能需要更长时间。

### Q: 如何处理意见不统一？

A: 通过讨论达成共识，必要时寻求第三方意见。

### Q: 可以跳过审查吗？

A: 不可以，所有代码都需要经过审查。
