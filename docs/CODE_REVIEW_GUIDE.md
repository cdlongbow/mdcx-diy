# 代码审查指南

本文档说明 MDCx 项目代码审查的流程、方法和规范。

## 代码审查流程

### 1. 创建 Pull Request

提交代码给别人审查前：

1. 确认代码能在本地跑通
2. 运行 `uv run ruff check .` 检查代码格式
3. 运行 `uv run pytest tests/` 跑一遍测试
4. 更新相关的文档
5. 写清楚 PR 描述

### 2. 审查角度

- **功能**：代码实现了想要的功能吗
- **质量**：代码清楚吗，好维护吗
- **性能**：有没有性能问题
- **安全**：有没有安全漏洞
- **测试**：测试覆盖了主要场景吗
- **文档**：文档更新了吗

### 3. 给反馈

- 说具体问题
- 提建设性意见
- 给出改进建议

### 4. 处理反馈

- 认真看完所有意见
- 不懂就问
- 尽快修改

## 审查最佳实践

### 给审查者

1. **及时看**：最好 24 小时内完成审查
2. **全面看**：整体看代码，不只盯局部
3. **多鼓励**：好的地方也说出来

### 给代码作者

1. **小步提交**：一次 PR 不要改太多东西
2. **说清楚**：PR 描述要详细
3. **快回复**：别人提了意见要及时回应

## 审查工具

### 自动化检查

- **ruff**：检查代码格式
- **pytest**：跑单元测试
- **pytest-cov**：测试覆盖率
- **CI/CD**：自动跑这些检查

### 手动检查

- **人工走查**：一行一行看代码
- **手动测试**：自己跑一下功能
- **性能测试**：检查性能瓶颈
- **安全审查**：检查安全漏洞

## 审查模板

### PR 描述模板

```markdown
## 改了啥

这次改了什么、为什么改

## 改的类型

- [ ] 新功能
- [ ] 修 Bug
- [ ] 重构代码
- [ ] 更新文档
- [ ] 优化性能
- [ ] 其他

## 影响范围

列出来哪些模块和文件受到了影响

## 怎么测试

说说其他人怎么验证你的修改

## 关联的 Issue

贴一下相关的 Issue 或 PR 链接

## 截图（如果有）

放几张截图让人更直观地看到效果
```

### 审查评论模板

```markdown
### 问题

说清楚你发现的问题是什么

### 建议

告诉人家可以怎么改

### 代码示例（如果有）

```python
# 举个具体的例子
```
```

## 审查标准分类

### 必须修（Must Fix）

不改不能合并：

- 功能写错了
- 安全有问题
- 性能太差
- 测试跑不过

### 建议修（Should Fix）

最好改一下：

- 代码风格不一致
- 代码难读懂
- 文档没更新
- 测试不够全

### 可修可不修（Nice to Fix）

改更好，不改也行：

- 变量名可以更贴切
- 注释可以写得更好
- 小地方可以优化

## 常见问题

### Q: 审查一次要多久？

A: 改得少的话 1-2 小时，改得多可能需要更久。

### Q: 审查者和作者意见不一致怎么办？

A: 先讨论，争取达成一致。实在不行找第三方来评理。

### Q: 可以不审查就直接合代码吗？

A: 不行。所有代码都必须经过审查才能合并。
