# 代码审查指南

这份文档告诉你 MDCx 项目怎么做代码审查。新手也能看懂。

## 代码审查流程

### 1. 创建 Pull Request

提交代码给别人审查之前，先做这些准备：

1. 确认代码能在本地跑通
2. 运行 `uv run ruff check .` 检查代码格式
3. 运行 `uv run pytest tests/` 跑一遍测试
4. 更新相关的文档
5. 写清楚 PR 描述，让别人一看就懂改了啥

### 2. 审查时要看什么

审查的人要从这几个角度去看：

- **功能对不对**：代码真的实现了想要的功能吗
- **写得好不好**：代码清楚吗，以后好维护吗
- **跑得快不快**：有没有性能问题
- **安不安全**：有没有安全漏洞
- **测试够不够**：测试覆盖了主要场景吗
- **文档全不全**：文档有没有跟着更新

### 3. 怎么给反馈

给反馈的时候要注意：

- 说具体问题，别只说"这不对"
- 提建设性意见，最好告诉人家怎么改
- 态度要好，尊重别人的工作
- 给出改进建议，不只挑毛病

### 4. 怎么处理反馈

收到反馈的人应该：

- 认真看完所有意见
- 看不懂的地方就问
- 尽快把问题修掉
- 心态开放，别抵触

## 审查最佳实践

### 给审查者的建议

1. **及时看**：最好 24 小时内完成审查
2. **全面看**：别只看部分代码，要整体看
3. **多鼓励**：看到好的地方也说出来
4. **有礼貌**：大家都是想把事情做好

### 给代码作者的建议

1. **小步提交**：一次 PR 不要改太多东西
2. **说清楚**：PR 描述要详细，让人好理解
3. **快回复**：别人提了意见要及时回应
4. **持续进步**：每次审查都在帮你把代码写得更好

## 审查工具

### 自动化检查

- **ruff**：检查代码格式对不对
- **pytest**：跑单元测试
- **pytest-cov**：看测试覆盖了多少代码
- **CI/CD**：自动帮你跑这些检查

### 手动检查

- **人工走查**：一行一行看代码
- **手动测试**：自己跑一下功能正不正常
- **性能测试**：看看有没有跑得慢的地方
- **安全审查**：检查有没有安全漏洞

## 审查模板

### PR 描述模板

```markdown
## 改了啥

简单说下这次改了什么、为什么改

## 改的类型

- [ ] 新功能
- [ ] 修 Bug
- [ ] 重构代码
- [ ] 更新文档
- [ ] 优化性能
- [ ] 其他

## 影响范围

列出来哪些模块和文件受到了影响

## 怎么测试

说说其他人怎么验证你的修改

## 关联的 Issue

贴一下相关的 Issue 或 PR 链接

## 截图（如果有）

放几张截图让人更直观地看到效果
```

### 审查评论模板

```markdown
### 问题

说清楚你发现的问题是什么

### 建议

告诉人家可以怎么改

### 代码示例（如果有）

```python
# 举个具体的例子
```
```

## 审查标准分类

### 必须修（Must Fix）

不改不能合并：

- 功能写错了
- 安全有问题
- 性能太差
- 测试跑不过

### 建议修（Should Fix）

最好改一下：

- 代码风格不一致
- 代码难读懂
- 文档没更新
- 测试不够全

### 可修可不修（Nice to Fix）

改更好，不改也行：

- 变量名可以更贴切
- 注释可以写得更好
- 小地方可以优化

## 常见问题

### Q: 审查一次要多久？

A: 改得少的话 1-2 小时，改得多可能需要更久。

### Q: 审查者和作者意见不一致怎么办？

A: 先讨论，争取达成一致。实在不行找第三方来评理。

### Q: 可以不审查就直接合代码吗？

A: 不行。所有代码都必须经过审查才能合并。
